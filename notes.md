# Notes

> this file is for the user to edit. Only add notes when the user asks

- ASP.NET MVC (`dotnet new mvc`) installs jQuery and Bootstrap by default
- `Properties/launchSettings.json` configures how the app launches through Visual Studio. We use dip for that instead
- `Bookshelf.csproj` is like a `tsconfig.json` or `package.json` â€” project configuration for development (target framework, compiler settings, package dependencies)
- `appsettings.json` configures the runtime of the app (logging, connection strings, etc.). Environment-specific overrides go in `appsettings.Development.json`
- `Program.cs` is sort of like the rackup in Rails â€” it's the entry point that configures services, middleware, and routes, then starts the server
- Routes can be defined inline on actions in controllers (via `[Route]` attributes) or in `Program.cs` (via `MapControllerRoute`). There's no separate routes file
- View lookup is convention-based: `return View()` in `HomeController.Index()` renders `Views/Home/Index.cshtml`. No explicit wiring needed. The lookup also passes through `_ViewStart.cshtml` (where the shared `_Layout.cshtml` is set) and `_ViewImports.cshtml` (where common namespaces and tag helpers are imported)
- No per-request logging by default. `UseHttpLogging()` enables it but it's verbose HTTP wire-level output, not concise Rails-style request logs
- Type-safe data passing to views happens by passing a model object to the `View()` constructor (e.g. `return View(book)`). `ViewData`/`ViewBag` exist but are untyped dictionaries. There's no implicit passing of data â€” everything goes through `View()` or `ViewData`/`ViewBag` explicitly. Only one model can be passed to a view, which leads to composing data into dedicated view model classes
- Third-party dependencies come from NuGet (nuget.org) and are managed as `<PackageReference>` entries in the `.csproj` file. There is no dependency pinning by default â€” no lock file is generated unless you opt in with `<RestorePackagesWithLockFile>true</RestorePackagesWithLockFile>` in the csproj. This feels bad from a supply chain security standpoint. When enabled, `packages.lock.json` records exact resolved versions and content hashes
- ASP.NET uses a thread pool with async-based thread sharing. A request is **not** locked to a single thread â€” when it hits an `await` (DB query, HTTP call, etc.), the thread is released back to the pool. This means you can't use thread-local globals for request state. On the upside, you don't need to worry about thread capacity as much: each request only occupies a thread while doing CPU-bound work, not during I/O-bound work. This is fundamentally different from Rails' process/thread concurrency model
- ASP.NET has no equivalent to `rails console`. There's no built-in REPL connected to your running app, DbContext, or dependency injection container. `dotnet-script`/`csi` gives you a standalone C# REPL but it's more like `irb` than `rails console`. The expected workflow is to use an IDE debugger (Visual Studio/Rider) with breakpoints and the Immediate Window, or write unit tests. You could DIY a rails-console-like experience by creating a small console app project that references your main project and bootstraps the same services, but nothing comes out of the box
- Migrations are automatically handled by `dotnet ef migrations add <Name>`. Make changes to your model files and the migrations will be generated automatically
- Associations have to be explicitly loaded (via `Include()`) to be accessed. Accessing an unloaded association returns null, which isn't an indication that the association isn't there â€” it just might not have been loaded
- Binding entity models directly to forms causes MVC validation errors because non-nullable navigation properties (like `Book.Author`) fail validation even when the foreign key is set. Most forms should bind to a ViewModel instead â€” a plain class with only the form fields â€” then map to the entity in the controller
- Migration `*.Designer.cs` files capture the model state at the time of each migration. EF uses them to diff against when generating the next migration and to support reverting. `ApplicationDbContextModelSnapshot.cs` is the cumulative current state of the model as EF understands it (like `db/schema.rb` in Rails) â€” it gets updated with each new migration
- Routes in views are generated using ASP Tag Helpers â€” `asp-controller`, `asp-action`, `asp-route-id`, etc. These are `asp-` prefixed attributes on HTML elements that Razor compiles into proper URLs. `asp-for` on form elements is strongly typed against the view's `@model`, but `asp-controller` and `asp-action` are just strings with no compile-time verification
- Custom tag helpers let you create reusable HTML components. You write a C# class extending `TagHelper`, use `[HtmlTargetElement]` to define the element name, and inject `IHtmlGenerator` to produce standard ASP.NET form elements. Register them in `_ViewImports.cshtml` with `@addTagHelper *, YourAssemblyName`. This is how we DRY up repeated form field markup (label + input + validation) into a single `<form-field asp-for="..." />` element
- Partial views work like Rails partials. Create a `.cshtml` file (conventionally `_`-prefixed, e.g. `_Navigation.cshtml`) in `Views/Shared/` and render it with `<partial name="_Navigation" />` (tag helper syntax) or `@await Html.PartialAsync("_Navigation")` (async helper syntax). Both are server-side only â€” the full HTML is composed on the server and sent in a single response. The tag helper internally calls the async path anyway, so for simple markup-only partials they're identical. `Html.PartialAsync` is worth using explicitly when a partial does I/O (e.g. database queries via a view component) so the thread is released back to the pool during the wait. Partials inherit the parent's model/ViewData by default, or you can pass a specific model via `<partial name="_Foo" model="someModel" />`
- ASP.NET has no image/file upload framework out of the box â€” you have to build your own. This project has a direct upload system built in
- Controller inheritance is a bit nicer here for building little controller mini-frameworks because C#'s type system can enforce that inheriting controllers provide the right shape (interfaces/base classes/generic constraints), rather than relying on convention or runtime assumptions
- Controllers inherit a `User` property (`ClaimsPrincipal`) from the `Controller` base class, populated by the auth middleware. `User.FindFirstValue(ClaimTypes.NameIdentifier)` reads the current user's id from the auth cookie claims â€” no DB round-trip. This is the simplest way to get the current user id in a controller. For repeated use, a helper property like `private string CurrentUserId => User.FindFirstValue(ClaimTypes.NameIdentifier)!;` keeps things clean. If you ever need the current user id outside a controller (e.g. a service or middleware), you'd need to inject `IHttpContextAccessor` to reach the `HttpContext.User` â€” but don't add that abstraction until you actually need it
- URL-hack defense pattern: when loading a single row for Details/Edit/Delete, always AND the ownership check into the query itself (`FirstOrDefaultAsync(b => b.Id == id && b.UserId == userId)`) rather than loading by id first and then checking ownership in an `if`. This makes "not yours" indistinguishable from "doesn't exist" (both return NotFound), prevents timing-based probing, and is one less branch to forget. Same idea as `current_user.books.find(params[:id])` in Rails
- EF Core has a Global Query Filters feature (`HasQueryFilter` in `OnModelCreating`) that can automatically add `.Where(b => b.UserId == currentUserId)` to every query on an entity. We chose NOT to use it because: (1) the admin area needs unfiltered global access, so you'd immediately need `IgnoreQueryFilters()` on every admin query, (2) it's invisible â€” a developer reading a controller wouldn't see the filter and might not realize it's there, (3) it makes debugging harder when rows "disappear" from queries for non-obvious reasons. Explicit `.Where()` in each controller is more verbose but leaves no mystery about what's being queried
- Authorâ†’Book cascade delete (`ON DELETE CASCADE` on `FK_Books_Authors_AuthorId`) is silent and data-destructive: deleting an author in the UI deletes all their books with no confirmation beyond the delete page's warning. The same applies to the Userâ†’Books and Userâ†’Authors cascades added for per-user ownership. Because these cascades happen at the database level, EF's change tracker never sees the cascaded deletes â€” meaning `SaveChangesAsync` overrides (like our file cleanup logic) won't fire for the cascaded rows. Orphaned cover image files from cascade-deleted books would need to be caught by the background `OrphanedUploadCleanupJob`
- `dotnet watch` hot reload does not reliably pick up changes to controller attributes (`[Authorize]`, `[AllowAnonymous]`, etc.) or constructor signatures. The method body gets patched but the endpoint routing metadata â€” which is built at startup from the attributes â€” stays stale. When you change authorization attributes or DI wiring, restart the container (`docker compose restart web`) rather than relying on hot reload. The logs will say "Hot reload of changes succeeded" even when the metadata is stale, so you can't trust that message for attribute changes
- EF Core relationships require two properties where Rails uses one `belongs_to`. A scalar FK (`public string UserId { get; set; } = string.Empty;`) maps to the database column, and a navigation property (`public IdentityUser User { get; set; } = null!;`) lets you walk the relationship in code via `.Include(b => b.User)`. The `= string.Empty` and `= null!;` defaults are compiler noise â€” they satisfy the non-nullable warnings from NRT but are immediately overwritten by EF (loading from DB) or your code (setting before save). The navigation is only populated when you explicitly `.Include()` it; accessing it without Include returns null, not a lazy-loaded object like Rails' `belongs_to`
- ASP.NET Core uses constructor injection for dependencies. You don't pass services to controllers manually â€” the DI container inspects the constructor's parameter types, resolves each one from its registrations in `Program.cs`, and constructs the controller automatically on each request. This replaces Rails' pattern of inheriting helper methods (like `current_user`) from base classes. The dependency is declared in the constructor signature, the framework provides it. `builder.Services.AddScoped<IFoo, Foo>()` in `Program.cs` is the registration; `public MyController(IFoo foo)` is the consumption
- DI lifetime (singleton / scoped / transient) controls how long an injected object lives. Singleton = one instance for the entire app (like a constant set in a Rails initializer). Scoped = one instance per HTTP request (like `Current.user` or how ActiveRecord checks out a DB connection per request). Transient = a brand new instance every time it's requested. Most app services are scoped. `DbContext` is scoped. Stateless utilities like file storage can be singleton. Getting this wrong (e.g. injecting a scoped service into a singleton) causes subtle bugs where request-specific state leaks across requests
- .NET uses the interface + implementation pattern (`ICurrentUserService` + `CurrentUserService`) because C#'s static typing locks constructor parameters to exact types. An interface lets you swap implementations (real vs fake, HTTP vs background job) without changing every call site. Ruby's duck typing makes this unnecessary â€” any object that responds to the right methods works. In .NET, the interface is the "duck typing" equivalent: it defines the shape, and the DI container decides which concrete class to hand out. The cost is one extra file per service; the payoff is explicit, compiler-checked contracts
- Business validation that requires DB queries (e.g. "does this author belong to this user?") lives at the controller or service layer in .NET, not on the entity. This feels wrong coming from Rails where `validate :something` on the model is the standard approach. The difference is architectural: Rails entities (Active Record) have database access built in, so model-level validation can query freely. .NET entities are plain data objects (POCOs) with no knowledge of the database â€” they can't inject a DbContext or run queries. DB-dependent validation therefore goes in the controller or a service that has access to the DbContext. This is the standard .NET pattern, not a shortcut
